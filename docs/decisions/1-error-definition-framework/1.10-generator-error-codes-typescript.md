# 1.10 Generator `error-codes.ts`

[← Kembali ke 1. Error Definition Framework](./README.md)

> **TL;DR:** frontend menerima constant dan type error code yang dihasilkan langsung dari PHP sehingga tidak perlu menyalin daftar code secara manual.

## Tujuan

Menetapkan pipeline generation bersama, bentuk file TypeScript, command, output path, dan perilaku penulisan artifact.

## Keputusan

### Shared Generation Pipeline

Satu Artisan command menghasilkan TypeScript error codes dan JSON catalog:

```shell
php artisan error-definition:generate
```

Command menjalankan proses berikut secara berurutan:

1. menemukan enum melalui `ErrorDefinitionDiscovery`;
2. menjalankan rule Error Definition `EDF001–EDF008`;
3. menghentikan proses apabila terdapat lint error;
4. membaca setiap enum case melalui `ErrorDefinitionReader`;
5. mengurutkan seluruh `ResolvedErrorDefinition` berdasarkan code;
6. membentuk content TypeScript dan JSON di memory;
7. menulis artifact yang content-nya berubah.

Validation mapping `VAL001–VAL009` tidak menjadi gate generator karena artifact hanya bergantung pada Error Definition. Full validation audit tetap dijalankan melalui `error-definition:lint`.

Discovery, lint, resolve, atau generation failure menghasilkan exit code `1`. Definition yang gagal tidak dilewati dan partial catalog tidak dihasilkan secara sengaja.

Generation hanya berjalan melalui command atau direct tooling call. Service Provider tidak menjalankannya ketika aplikasi boot.

### Output Path

Default output path ditetapkan melalui `config/error-definition.php`:

```php
return [
    'generation' => [
        'typescript_path' => resource_path(
            'js/generated/error-codes.ts',
        ),
        'catalog_path' => storage_path(
            'app/error-definition/error-catalog.json',
        ),
    ],
];
```

Path dapat diarahkan ke frontend repository atau build workspace lain. Parent directory dibuat otomatis apabila belum tersedia.

### Generation Lifecycle

TypeScript dan JSON merupakan generated artifact dan tidak disimpan dalam Git secara default. Error Definition PHP tetap menjadi satu-satunya source of truth.

Pada local development, developer menjalankan command sesuai kebutuhan setelah Error Definition berubah atau sebelum menjalankan frontend yang membutuhkan artifact terbaru.

CI menjalankan normal generation dari source revision yang sedang dibangun, setelah PHP dependency tersedia dan sebelum frontend build atau publication catalog. Generation failure menghentikan pipeline. Production instance menggunakan hasil build dan tidak menjalankan generator ketika boot atau menerima request.

Project yang secara eksplisit menyimpan generated artifact dalam Git dapat menggunakan `--check`, tetapi workflow tersebut bukan default framework.

### Penulisan Artifact

Seluruh content dibentuk dan divalidasi sebelum proses write dimulai. Setiap artifact ditulis ke temporary file pada directory yang sama lalu dipindahkan secara atomic ke target.

File hanya diganti ketika generated content berbeda dari file yang tersedia. Perilaku ini mempertahankan modification time dan build cache ketika tidak ada perubahan Error Definition.

Tidak ada intermediate definition file atau persistent generation cache. Process yang sama menggunakan in-memory result dari discovery dan Reader.

Generated content tidak memuat timestamp. Input yang sama harus menghasilkan byte content yang sama pada operating system berbeda.

### Check Mode

Command menyediakan mode pemeriksaan tanpa menulis file:

```shell
php artisan error-definition:generate --check
```

Missing atau stale artifact menghasilkan exit code `1`. Artifact yang sama dengan generated content menghasilkan exit code `0`. Mode ini tersedia bagi project yang memilih menyimpan generated artifact dalam Git.

### TypeScript Contract

File `error-codes.ts` menggunakan `as const` object dan exported union type:

```ts
// This file is generated by Error Definition Framework.
// Do not edit manually.

export const ERROR_CODES = {
  'SM_VAL_001': 'SM-VAL-001',
  'SM_WF_001': 'SM-WF-001',
  'USR_BR_001': 'USR-BR-001',
} as const;

export type ErrorCode =
  (typeof ERROR_CODES)[keyof typeof ERROR_CODES];
```

Property key dibentuk dengan mengganti `-` pada error code menjadi `_`. Transformasi tersebut tidak menghasilkan collision karena format code tidak mengizinkan underscore dan linter menjamin error code unik.

Property key selalu ditulis sebagai string literal agar tetap valid apabila segment code dimulai dengan angka. Standard code seperti `SM_WF_001` tetap dapat diakses menggunakan dot notation:

```ts
if (error.code === ERROR_CODES.SM_WF_001) {
  // ...
}
```

TypeScript artifact hanya membawa error code. Message, category, HTTP status, severity, dan retryable tersedia pada JSON catalog dan tidak diduplikasi ke file TypeScript.

Generator tidak menghasilkan TypeScript `enum`, helper function, PHP FQCN, enum case name, atau source comment. Frontend hanya bergantung pada public error code.

Formatting menggunakan dua spasi, single quote, semicolon, LF line ending, dan satu newline pada akhir file. Generator tidak bergantung pada Prettier atau formatter frontend.

## Alasan

Satu command menjaga kedua artifact berasal dari discovery dan resolved definitions yang sama. Command terpisah berisiko membuat TypeScript dan JSON diperbarui pada waktu berbeda.

Linter dijalankan sebelum Reader agar seluruh kesalahan definition ditampilkan bersama. Validation mapping tidak menjadi dependency karena kesalahannya tidak mengubah daftar maupun metadata error yang dihasilkan.

`as const` object menyediakan runtime constant dan compile-time union tanpa runtime behavior tambahan dari TypeScript enum. Property yang diturunkan dari code tetap unik meskipun beberapa PHP enum memiliki case bernama sama.

Metadata tidak dimasukkan ke TypeScript agar frontend tidak menerima dua bentuk catalog. Frontend yang membutuhkan metadata dapat membaca JSON catalog atau response API sesuai kebutuhan.

Generated header mencegah file dianggap sebagai source manual. Tidak menyertakan FQCN menjaga frontend bebas dari namespace dan refactor internal PHP.

Write-if-changed menghindari rebuild serta diff yang tidak diperlukan. Temporary file dan atomic rename mencegah target berisi content setengah jadi ketika proses write gagal.

Timestamp tidak digunakan karena membuat artifact berubah pada setiap command meskipun source definition tetap sama.

Internal formatting membuat output deterministik tanpa mewajibkan Node.js, Prettier, atau dependency frontend pada backend command.

Generation pada CI memastikan artifact release selalu berasal dari source revision yang sama. Developer yang lupa menjalankan generator secara lokal tidak dapat menghasilkan release dengan artifact lama.

Artifact tidak disimpan dalam Git secara default karena keduanya dapat dibuat ulang secara deterministik dari Error Definition PHP.

## Konsekuensi

- Frontend memperoleh runtime constant dan union type dari source PHP yang sama.
- Error code menjadi dasar nama property TypeScript.
- Perubahan error code mengubah public TypeScript contract.
- Generated file tidak membawa metadata atau detail source PHP.
- CI menghasilkan kedua artifact sebelum frontend build dan publication catalog.
- Developer tidak perlu menyertakan generated artifact dalam commit.
- Mode `--check` tetap tersedia bagi project yang memilih versioned artifact.
- Aplikasi yang memisahkan frontend dapat mengarahkan output path melalui config.
- Kedua artifact dihasilkan melalui command yang sama.
