# 1.11 Generator `error-catalog.json`

[← Kembali ke 1. Error Definition Framework](./README.md)

> **TL;DR:** seluruh metadata runtime error diekspor menjadi catalog JSON yang stabil dan siap dipakai tooling maupun Central Error Registry.

## Tujuan

Menetapkan schema, field, serialization, dan batas informasi pada generated JSON catalog.

## Keputusan

### Bentuk Catalog

`error-catalog.json` menggunakan top-level object yang memiliki integer `schemaVersion` dan array `errors`:

```json
{
  "schemaVersion": 1,
  "errors": [
    {
      "code": "SM-WF-001",
      "message": "Draft sedang dikunci.",
      "category": "workflow",
      "httpStatus": 409,
      "severity": "medium",
      "retryable": false
    }
  ]
}
```

`schemaVersion` dimulai dari `1` dan berubah ketika bentuk atau makna field catalog mengalami breaking change. Nilai ini bukan versi package atau versi aplikasi.

Raw array tidak digunakan karena tidak menyediakan tempat stabil untuk schema version dan metadata catalog pada masa depan.

### Field Error

Setiap item membawa seluruh metadata runtime dari `ResolvedErrorDefinition`:

- `code`;
- `message`;
- `category`;
- `httpStatus`;
- `severity`;
- `retryable`.

Category dan severity menggunakan backing value enum, bukan nama PHP enum case. Field menggunakan camel case agar konsisten sebagai kontrak JSON.

Message disimpan sebagai default template. Placeholder seperti `:attribute`, `:min`, dan `:max` tidak diisi dengan example value karena nilainya baru tersedia ketika validation dijalankan.

Errors diurutkan secara alfabetis berdasarkan `code`.

### Batas Informasi

Catalog tidak memuat:

- PHP FQCN atau enum case name;
- application name atau application version;
- generation timestamp;
- owner, SLA, escalation, atau Knowledge Base;
- registry enrichment lain.

Source identity tidak disertakan karena `ResolvedErrorDefinition` sengaja tidak membawa detail enum atau Reflection. Error code menjadi identifier yang menghubungkan catalog dengan aplikasi dan registry.

Application identity ditetapkan bersama kontrak publish pada Stage 2. Metadata organisasional tetap dikelola oleh registry sesuai Decision 1.1.1.

### Serialization

JSON menggunakan pretty print dua spasi, Unicode dan slash tanpa escape yang tidak diperlukan, LF line ending, serta satu newline pada akhir file.

Serialization setara dengan opsi berikut:

```php
JSON_PRETTY_PRINT
| JSON_UNESCAPED_SLASHES
| JSON_UNESCAPED_UNICODE
| JSON_THROW_ON_ERROR
```

Encoding failure menghentikan command dengan exit code `1`. Generator tidak menghasilkan compact JSON atau partial catalog.

### Generation dan Publication

Catalog menggunakan shared pipeline, command, config path, write-if-changed, atomic write, dan `--check` mode yang ditetapkan pada [Decision 1.10](./1.10-generator-error-codes-typescript.md).

Catalog dibuat sebagai local build artifact dan tidak disimpan dalam Git secara default. CI menghasilkannya dari source revision yang sedang dibangun sebelum publication.

Generator hanya membuat local artifact. Upload, authentication, application identity, schema validation oleh server, dan publication ke Central Error Registry menjadi bagian Stage 2.

## Alasan

Versioned top-level object memungkinkan catalog berkembang tanpa menebak bentuk data dari content. Integer cukup karena schema version hanya berubah ketika consumer perlu memahami kontrak baru.

Seluruh metadata runtime disertakan agar tooling tidak perlu membaca source PHP. Field organisasional tidak disediakan sebagai `null` karena nilainya bukan tanggung jawab aplikasi dan dapat berubah tanpa deployment.

Backing value dipilih sebagai public value yang stabil. Nama enum case merupakan detail source PHP dan dapat berbeda dari nilai yang dipertukarkan antar-sistem.

Placeholder dipertahankan agar catalog merepresentasikan Error Definition, bukan satu contoh validation failure.

Tidak menyertakan timestamp membuat output deterministik. Perubahan file selalu menunjukkan perubahan schema atau Error Definition yang nyata.

Pretty-printed JSON dipilih karena catalog dapat dibaca manusia, ditinjau melalui Git diff, dan tetap dapat diproses oleh parser standar.

Publication dipisahkan dari generation agar aplikasi dapat membuat serta memvalidasi artifact tanpa network dependency. Kegagalan registry tidak memengaruhi kemampuan aplikasi menghasilkan catalog lokal.

## Konsekuensi

- `schemaVersion` menjadi kontrak bagi seluruh catalog consumer.
- Catalog membawa metadata runtime lengkap tanpa enrichment registry.
- Perubahan field atau maknanya dapat membutuhkan schema version baru.
- Source namespace dan enum case tidak bocor ke consumer.
- Catalog dapat dibuat tanpa koneksi ke Central Error Registry.
- Catalog dibuat ulang oleh CI untuk setiap build yang membutuhkannya.
- Publication dan application identity tetap menjadi keputusan Stage 2.
