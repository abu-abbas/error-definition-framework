# 1.2 Struktur Enum Error per Modul

[← Kembali ke 1. Error Definition Framework](./README.md)

> **TL;DR:** error ditempatkan pada konteks bisnis yang memiliki aturannya, bukan dikumpulkan berdasarkan halaman tempat error terlihat.

## Tujuan

Menetapkan cara mengelompokkan error code berdasarkan modul, fitur, atau proses bisnis.

## Keputusan

Satu enum mewakili satu konteks bisnis yang memiliki vocabulary, aturan, atau lifecycle yang dapat dijelaskan secara mandiri. Framework tidak menggunakan satu enum global seperti `ApplicationError` untuk seluruh aplikasi.

Pemilik error ditentukan berdasarkan aturan atau proses yang gagal, bukan berdasarkan controller, form, atau halaman yang menampilkan error.

Error lintas modul memiliki enum tersendiri hanya jika relasinya membentuk proses bisnis dengan aturan sendiri.

## Alasan

Enum global akan terus membesar, mencampur ownership, dan menyulitkan pencarian error. Pengelompokan berdasarkan konteks bisnis membuat perubahan pada satu area tidak mengaburkan error milik area lain.

Batas konteks ditentukan dari sumber aturannya:

- Jika error menjelaskan keadaan satu entitas, error dimiliki enum entitas tersebut.
- Jika error menjelaskan aturan interaksi antara beberapa entitas, interaksi tersebut dapat memiliki enum tersendiri.
- Halaman atau form bukan konteks baru jika hanya menampilkan aturan yang sudah dimiliki modul lain.

✅ **Contoh:** role sudah dimiliki user → `UserRoleAssignmentError`, karena proses assignment memiliki aturan relasi antara User dan Role.

❌ **Bukan contoh:** email user tidak valid pada halaman pengaturan role → tetap `UserError`, karena aturan validitas email dimiliki User; halaman tempat error muncul tidak memindahkan ownership.

✅ **Contoh:** proses persetujuan Surat Masuk memiliki status dan transisi sendiri → tetap bagian dari `SuratMasukError` selama vocabulary dan lifecycle-nya masih milik Surat Masuk.

❌ **Bukan contoh:** setiap tahap workflow dibuat menjadi enum berbeda seperti `DraftError` dan `ApprovalError` hanya karena tampil pada halaman berbeda; pemisahan tersebut tidak menambah ownership bisnis yang baru.

## Konsekuensi

- Error code lebih mudah ditemukan berdasarkan konteksnya.
- Ownership error dapat ditentukan dari sumber aturan bisnis.
- Relasi lintas modul tidak otomatis menghasilkan enum baru.
- Aplikasi dapat memilih struktur folder tanpa mengubah kontrak framework.
- Discovery dan generator tidak boleh mengasumsikan lokasi enum tertentu.
