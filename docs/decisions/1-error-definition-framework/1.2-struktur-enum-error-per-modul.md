# 1.2 Struktur Enum Error per Modul

[← Kembali ke 1. Error Definition Framework](./README.md)

## Tujuan

Error code dikelompokkan berdasarkan modul, fitur, atau proses bisnis yang memiliki aturan terkait.

Dokumen ini menetapkan cara membagi enum error berdasarkan konteksnya. Struktur folder aplikasi dibahas sebagai opsi implementasi, bukan sebagai kontrak framework.

## Keputusan Utama

Satu enum mewakili satu modul, fitur, atau proses bisnis yang cukup jelas. Error tidak digabungkan ke dalam satu enum global.

```php
enum ApplicationError: string
{
    // seluruh error aplikasi
}
```

Pendekatan tersebut tidak dipilih karena enum global akan membesar seiring pertumbuhan aplikasi dan menyulitkan pemeliharaan.

## Definisi Enum Error

Enum error menjadi source of truth untuk error code. Metadata error ditempelkan pada setiap enum case menggunakan `ErrorDefinition`.

```php
enum SuratMasukError: string implements ErrorCode
{
    #[ErrorDefinition(
        message: 'Nomor surat wajib diisi.',
        category: ErrorCategory::VALIDATION,
        httpStatus: 422,
        severity: ErrorSeverity::LOW,
    )]
    case NOMOR_SURAT_REQUIRED = 'SM-VAL-001';

    #[ErrorDefinition(
        message: 'Surat sudah disetujui dan tidak dapat diubah.',
        category: ErrorCategory::WORKFLOW,
        httpStatus: 409,
        severity: ErrorSeverity::MEDIUM,
    )]
    case DRAFT_LOCKED = 'SM-WF-001';
}
```

Setiap enum case membawa:

- **Error Code**, yang berasal dari nilai Backed Enum;
- **ErrorCategory**, yang menyatakan jenis atau konteks penyebab error;
- **ErrorSeverity**, yang menyatakan tingkat dampak error;
- metadata lain seperti message, HTTP status, dan retryability.

`ErrorCategory` dan `ErrorSeverity` menggunakan enum, bukan string bebas. Keduanya disimpan sebagai metadata pada `ErrorDefinition`, bukan dikodekan ulang pada nama enum atau prefix error code.

## Nilai Minimum `ErrorCategory`

`ErrorCategory` menjelaskan jenis atau konteks penyebab error. Category tidak menunjukkan tingkat dampak error.

```php
enum ErrorCategory: string
{
    case VALIDATION = 'validation';
    case AUTHENTICATION = 'authentication';
    case AUTHORIZATION = 'authorization';
    case NOT_FOUND = 'not_found';
    case BUSINESS_RULE = 'business_rule';
    case WORKFLOW = 'workflow';
    case INTEGRATION = 'integration';
    case SYSTEM = 'system';
}
```

| Category | Penggunaan |
|----------|------------|
| `VALIDATION` | Input tidak memenuhi aturan validasi. |
| `AUTHENTICATION` | Identitas pengguna tidak dapat diverifikasi. |
| `AUTHORIZATION` | Pengguna tidak memiliki hak untuk melakukan operasi. |
| `NOT_FOUND` | Resource atau data yang diminta tidak ditemukan. |
| `BUSINESS_RULE` | Operasi melanggar aturan bisnis. |
| `WORKFLOW` | Operasi tidak sesuai dengan status atau tahapan proses. |
| `INTEGRATION` | Komunikasi dengan sistem lain mengalami masalah. |
| `SYSTEM` | Terjadi kegagalan internal aplikasi atau infrastruktur. |

Category dipilih berdasarkan penyebab error, bukan berdasarkan halaman tempat error ditampilkan.

## Nilai Minimum `ErrorSeverity`

`ErrorSeverity` menjelaskan tingkat dampak error terhadap pengguna, proses bisnis, atau layanan.

```php
enum ErrorSeverity: string
{
    case LOW = 'low';
    case MEDIUM = 'medium';
    case HIGH = 'high';
    case CRITICAL = 'critical';
}
```

| Severity | Penggunaan |
|----------|------------|
| `LOW` | Dampak terbatas dan proses utama masih dapat dilanjutkan. |
| `MEDIUM` | Operasi tertentu gagal, tetapi layanan masih dapat digunakan. |
| `HIGH` | Fitur atau proses penting tidak dapat digunakan dan memerlukan perhatian. |
| `CRITICAL` | Layanan utama terganggu atau berpotensi menyebabkan dampak luas. |

Severity digunakan oleh logging, monitoring, dan analisis operasional. Severity tidak menentukan penyebab error.

Daftar category dan severity tersebut merupakan baseline framework. Nilai tambahan dapat ditambahkan sesuai kebutuhan aplikasi, tetapi makna nilai yang sudah digunakan tidak boleh diubah secara sembarangan.

## Format Error Code

Format error code menggunakan prefix modul atau fitur, kategori error, dan nomor urut.

```text
SM-VAL-001
│   │   └── nomor urut
│   └────── kategori error
└────────── prefix modul atau fitur
```

Contoh prefix:

```text
SM   = Surat Masuk
USR  = User
ROL  = Role
URA  = User Role Assignment
```

Prefix harus konsisten dalam satu konteks error dan tidak boleh digunakan oleh konteks lain secara ambigu.

## Penentuan Pemilik Error

Pemilik error ditentukan berdasarkan aturan atau proses yang gagal, bukan berdasarkan halaman tempat error ditampilkan.

| Kondisi | Enum yang sesuai |
|---------|------------------|
| Email user sudah digunakan | `UserError` |
| User tidak ditemukan | `UserError` |
| Role tidak ditemukan | `RoleError` |
| Role sudah dimiliki user | `UserRoleAssignmentError` |
| Role tidak dapat diberikan kepada user tertentu | `UserRoleAssignmentError` |

Error yang berasal dari relasi atau proses lintas modul dapat memiliki enum tersendiri apabila konteks bisnisnya berbeda dari enum modul asal.

```php
enum UserRoleAssignmentError: string implements ErrorCode
{
    #[ErrorDefinition(
        message: 'Role tersebut sudah dimiliki oleh pengguna.',
        category: ErrorCategory::BUSINESS_RULE,
        httpStatus: 409,
    )]
    case ROLE_ALREADY_ASSIGNED = 'URA-BR-001';

    #[ErrorDefinition(
        message: 'Role tidak dapat diberikan kepada pengguna ini.',
        category: ErrorCategory::BUSINESS_RULE,
        httpStatus: 422,
    )]
    case ROLE_NOT_ASSIGNABLE = 'URA-BR-002';
}
```

## Penggunaan pada Validation

Validation menggunakan enum error yang sama dengan komponen runtime lainnya. Validation tidak mendefinisikan error code atau message baru.

```php
public function errorCodes(): array
{
    return [
        'nomor_surat.required' => SuratMasukError::NOMOR_SURAT_REQUIRED,
    ];
}
```

Message validation diambil dari `ErrorDefinition`. `ErrorCategory` dan `ErrorSeverity` tetap tersedia melalui hasil resolve Reader sehingga validation, exception, dan response menggunakan definisi yang sama.

## Struktur Folder sebagai Opsi Implementasi

Struktur folder merupakan konvensi aplikasi, bukan kontrak framework. Package tidak boleh mengasumsikan bahwa enum selalu berada pada folder tertentu.

### Struktur Minimalis

Sesuai untuk aplikasi sederhana atau tim yang belum menerapkan pengelompokan modular.

```text
app/
└── Errors/
    ├── SuratMasukError.php
    ├── UserError.php
    ├── RoleError.php
    └── UserRoleAssignmentError.php
```

### Struktur Modular

Sesuai untuk aplikasi yang mengelompokkan kode berdasarkan modul.

```text
app/
└── Modules/
    ├── SuratMasuk/
    │   └── Errors/
    │       └── SuratMasukError.php
    ├── User/
    │   └── Errors/
    │       ├── UserError.php
    │       └── UserRoleAssignmentError.php
    └── Role/
        └── Errors/
            └── RoleError.php
```

### Struktur Berbasis Domain

Sesuai untuk aplikasi dengan aturan bisnis yang kompleks. Struktur ini tidak mensyaratkan penggunaan microservices dan tetap dapat digunakan pada Laravel monolith.

```text
app/
└── Domains/
    └── IdentityAccess/
        ├── Users/
        │   └── Errors/
        │       └── UserError.php
        ├── Roles/
        │   └── Errors/
        │       └── RoleError.php
        └── Assignments/
            └── Errors/
                └── UserRoleAssignmentError.php
```

Discovery atau generator tidak boleh bergantung pada struktur tersebut. Enum dapat didaftarkan melalui konfigurasi atau ditemukan melalui mekanisme discovery yang ditentukan kemudian.

## Aturan

- Satu enum mewakili satu modul, fitur, atau proses bisnis.
- Satu prefix digunakan secara konsisten untuk setiap enum atau konteks error.
- Nama enum case menjelaskan makna error, bukan isi pesannya.
- Nomor error yang telah digunakan tidak boleh digunakan ulang.
- Error dari relasi atau proses lintas modul dapat memiliki enum tersendiri.
- `ErrorCategory` dan `ErrorSeverity` menggunakan enum dan ditetapkan melalui `ErrorDefinition`.
- Validation menggunakan enum error yang sama dan tidak mendefinisikan message terpisah.
- Lokasi folder tidak ditentukan oleh framework.
- Struktur minimalis, modular, dan berbasis domain sama-sama didukung.
- Hindari satu enum besar seperti `ApplicationError`.

## Konsekuensi

- Error code lebih mudah ditemukan berdasarkan konteksnya.
- Pertumbuhan enum dapat dibatasi pada satu modul atau proses bisnis.
- Category dan severity tetap konsisten karena menggunakan enum.
- Validation, exception, dan response menggunakan definisi error yang sama.
- Aplikasi dapat mengadopsi struktur folder sesuai kebutuhan tanpa mengubah kontrak framework.
- Framework dan package tidak bergantung pada struktur folder aplikasi.

## Struktur Awal untuk Boilerplate

Untuk boilerplate, struktur minimalis digunakan sebagai pilihan awal karena paling mudah diikuti.

```text
app/Errors/
```

Aplikasi yang membutuhkan pengelompokan lebih lanjut dapat menggunakan struktur modular atau berbasis domain tanpa mengubah konsep Error Definition Framework.
