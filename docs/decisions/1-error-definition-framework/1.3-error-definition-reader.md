# 1.3 ErrorDefinitionReader

[ŌåÉ Kembali ke 1. Error Definition Framework](./README.md)

## Tujuan

Menetapkan tanggung jawab dan kontrak `ErrorDefinitionReader` sebagai komponen yang membaca metadata dari satu enum case dan menghasilkan `ResolvedErrorDefinition`.

Kontrak API tersedia pada [ErrorDefinitionReader Reference](../../reference/error-definition-reader.md). Bentuk DTO tersedia pada [ResolvedErrorDefinition Reference](../../reference/resolved-error-definition.md).

## 1.3.1 Responsibility

### Keputusan

`ErrorDefinitionReader` hanya bertanggung jawab membaca satu enum case, mengambil attribute `ErrorDefinition`, melakukan validasi dasar, dan mengembalikan metadata yang telah di-resolve.

### Di Luar Tanggung Jawab

Reader tidak bertanggung jawab terhadap:

- pemindaian folder atau module;
- penemuan seluruh enum error;
- pembuatan `error-catalog.json`;
- pembuatan `error-codes.ts`;
- validasi duplicate error code;
- validasi struktur project;
- logging;
- registry;
- knowledge base;
- monitoring;
- integrasi ITSM.

### Alasan

Reading dan discovery merupakan proses yang berbeda. Reading menerima satu enum case, sedangkan discovery menemukan seluruh Error Definition.

Discovery diperlukan oleh Error Linter, Error Catalog Generator, dan Central Error Registry. Menempatkannya di Reader akan memberikan terlalu banyak alasan perubahan pada satu class.

### Alternatif

Reader dapat sekaligus menyediakan method seperti berikut:

```php
$reader->discover();
$reader->scan(app_path());
```

Pendekatan tersebut tidak dipilih karena membuat Reader bergantung pada struktur folder dan mencampur proses discovery dengan reading.

### Konsekuensi

- Reader tetap kecil dan mudah diuji.
- Consumer tidak perlu mengetahui lokasi enum.
- Framework tidak bergantung pada struktur folder tertentu.
- Discovery dapat berkembang tanpa mengubah Reader.

## 1.3.2 Output

### Keputusan

Method `read()` mengembalikan `ResolvedErrorDefinition`, bukan `ErrorDefinition` secara langsung.

### Alternatif

`ErrorDefinition` hanya menyimpan metadata dari attribute. Error code berada pada enum. Jika object tersebut dikembalikan secara langsung, consumer harus membawa enum secara terpisah untuk memperoleh code.

Pendekatan tersebut tidak dipilih karena informasi yang secara konseptual merupakan satu Error Definition menjadi terbagi pada dua object.

### Alasan

`ResolvedErrorDefinition` menggabungkan code dari enum dengan metadata dari attribute. Consumer cukup menggunakan satu object untuk exception, validation, response, logging, dan generator.

### Konsekuensi

- API consumer lebih sederhana.
- Consumer tidak perlu membawa enum dan attribute secara bersamaan.
- Hasil resolve dapat langsung diserialisasi.
- Penambahan metadata tidak memerlukan perubahan pada consumer selama kontrak DTO tetap kompatibel.

## 1.3.3 Input

### Keputusan

Reader menerima enum yang mengimplementasikan marker interface `ErrorCode` dan merupakan `BackedEnum`.

### Alasan

`BackedEnum` saja terlalu umum karena juga mencakup enum status, role, atau tipe dokumen. Marker interface menyatakan bahwa enum tersebut memang dimaksudkan sebagai error code.

Kombinasi `ErrorCode` dan `BackedEnum` memberikan kontrak yang type-safe, self-documenting, dan tidak bergantung pada struktur folder.

### Alternatif: Method pada Interface

Interface dapat menambahkan method `code(): string`, tetapi pendekatan tersebut tidak dipilih karena kode sudah tersedia melalui `$error->value` pada Backed Enum. Method tambahan hanya menduplikasi kemampuan PHP.

### Alternatif: String

Input berupa string tidak dipilih karena menghilangkan informasi enum asal, pembacaan attribute, dan type checking.

### Konsekuensi

- Hanya enum error yang dapat diproses.
- Enum lain ditolak oleh type system.
- Reader tidak perlu mengetahui lokasi file enum.

## 1.3.4 MissingErrorDefinitionException

### Keputusan

Enum yang mengimplementasikan `ErrorCode` tetapi tidak memiliki attribute `ErrorDefinition` menghasilkan `MissingErrorDefinitionException`.

Reader tidak mengembalikan `null`, `false`, atau object kosong.

### Alasan

Implementasi `ErrorCode` menyatakan bahwa enum tersebut merupakan Error Definition yang valid. Ketiadaan attribute adalah kesalahan konfigurasi programmer, bukan business error.

Framework menerapkan fail fast agar kesalahan diketahui pada titik pertama kali ditemukan.

### Alternatif

Return `null` tidak dipilih karena setiap consumer harus melakukan null checking. Exception umum seperti `RuntimeException`, `LogicException`, atau `Exception` juga tidak dipilih karena tidak menjelaskan penyebab kegagalan secara spesifik.

### Konsekuensi

- Consumer tidak perlu melakukan null checking.
- Kesalahan konfigurasi langsung terlihat.
- Exception dapat ditangkap secara spesifik.
- Debugging menjadi lebih mudah.

## 1.3.5 Reflection dan In-Memory Cache

### Keputusan

Reader menggunakan PHP Reflection untuk membaca attribute dan menyimpan hasil resolve pada cache internal selama lifecycle Reader.

### Alasan

Reflection merupakan mekanisme standar PHP untuk membaca Attribute. Metadata error bersifat immutable sehingga proses Reflection dan resolve tidak perlu diulang untuk enum case yang sama.

### Data yang Disimpan

Cache menyimpan `ResolvedErrorDefinition`, bukan object Reflection atau `ErrorDefinition` mentah.

Menyimpan Reflection tetap mengharuskan Reader melakukan resolve pada setiap pemanggilan. Menyimpan `ErrorDefinition` juga membuat Reader membangun object hasil resolve berulang kali.

### Alternatif: Laravel Cache

Laravel Cache tidak dipilih karena metadata berukuran kecil, immutable, dan hanya diperlukan di dalam process. Cache internal tidak menambah dependency dan cukup untuk kebutuhan ini.

### Laravel Octane dan Long-Running Process

Cache aman digunakan pada Laravel Octane selama hanya menyimpan data immutable. Reader tidak boleh menyimpan Request, User, Locale, Tenant, Service, Exception, atau Runtime Context.

Perubahan source code pada Octane atau Queue Worker tetap membutuhkan restart process. Kondisi tersebut merupakan karakteristik long-running process, bukan masalah khusus cache Reader.

### Konsekuensi

- Reflection hanya dilakukan saat cache miss.
- Resolve hanya dilakukan satu kali untuk setiap enum case pada lifecycle Reader.
- Cache menggunakan sedikit memory karena metadata berukuran kecil.
- Implementasi Reader direkomendasikan menggunakan singleton agar cache tidak dibuat ulang.

## 1.3.6 Resolved DTO Tanpa Enum Source

### Keputusan

`ResolvedErrorDefinition` hanya menyimpan hasil resolve. DTO tidak menyimpan enum source, Reflection, `ReflectionAttribute`, `ReflectionEnum`, atau object Reflection lainnya.

### Alasan

Setelah proses resolve selesai, consumer hanya membutuhkan code, message, category, severity, retryable, dan HTTP status. Enum serta Reflection merupakan detail implementasi Reader dan tidak diperlukan oleh consumer.

### Konsekuensi

- DTO tetap sederhana.
- DTO mudah diserialisasi dan diuji.
- Consumer tidak bergantung pada Reflection.
- Consumer tidak bergantung pada enum source.
