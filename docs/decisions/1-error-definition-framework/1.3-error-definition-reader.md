# 1.3 ErrorDefinitionReader

[← Kembali ke 1. Error Definition Framework](./README.md)

## Tujuan

`ErrorDefinitionReader` membaca metadata error dari enum case yang mengimplementasikan `ErrorCode`, kemudian menggabungkan kode error dari enum dengan metadata dari attribute `ErrorDefinition` menjadi object yang siap digunakan oleh komponen framework.

Framework menggunakan dua sumber informasi:

- **PHP Backed Enum** sebagai sumber kode error.
- **PHP Attribute** sebagai sumber metadata error.

Kedua sumber informasi tersebut digabungkan menjadi `ResolvedErrorDefinition`. Dengan demikian, consumer tidak perlu mengetahui mekanisme penyimpanan metadata.

Pendekatan ini mempertahankan **Single Source of Truth** dan menyediakan metadata yang dapat digunakan oleh exception, validation, generator, maupun response builder.

---

# 1.3.1 Responsibility

## Keputusan

`ErrorDefinitionReader` hanya bertanggung jawab membaca satu enum case dan mengembalikan metadata error yang telah di-resolve.

```php
$definition = $reader->read(
    SuratMasukError::DRAFT_LOCKED
);
```

### Ruang Lingkup

Tanggung jawab `ErrorDefinitionReader` meliputi:

- menerima satu enum case;
- membaca attribute `ErrorDefinition`;
- melakukan validasi dasar;
- membangun `ResolvedErrorDefinition`;
- mengembalikan hasil resolve.

### Di Luar Tanggung Jawab

`ErrorDefinitionReader` tidak bertanggung jawab terhadap:

- pemindaian folder;
- penemuan seluruh enum error;
- pembuatan `error-catalog.json`;
- pembuatan `error-codes.ts`;
- validasi duplicate error code;
- validasi struktur project;
- logging;
- registry;
- knowledge base;
- monitoring;
- integrasi ITSM.

Kebutuhan tersebut merupakan tanggung jawab komponen lain dan dibahas pada roadmap berikutnya.

### Alasan

Pembatasan tanggung jawab diperlukan untuk menerapkan prinsip **Single Responsibility Principle (SRP)**. Discovery, generator, registry, dan validasi memiliki alasan perubahan yang berbeda dari proses pembacaan metadata.

Dengan membatasi tanggung jawab pada proses resolve metadata, Reader tetap sederhana, mudah diuji, dapat digunakan ulang, dan tidak bergantung pada implementasi lain.

### Alternatif

Reader dapat sekaligus melakukan discovery terhadap seluruh aplikasi.

```php
$reader->discover();
$reader->scan(app_path());
```

Pendekatan tersebut tidak dipilih karena discovery dan reading merupakan dua proses dengan tanggung jawab berbeda.

Reading menjawab pertanyaan mengenai cara membaca satu Error Definition, sedangkan discovery menjawab pertanyaan mengenai cara menemukan seluruh Error Definition.

Discovery diperlukan oleh Error Linter, Error Catalog Generator, dan Central Error Registry. Oleh karena itu, discovery ditempatkan pada komponen dan roadmap terpisah.

### Konsekuensi

- Reader hanya menangani satu Error Definition.
- Consumer tidak perlu mengetahui struktur project.
- Framework tidak bergantung pada struktur folder tertentu.
- Komponen discovery dapat dikembangkan tanpa mengubah Reader.

# 1.3.2 Output

## Keputusan

Method `read()` mengembalikan object `ResolvedErrorDefinition`.

```php
public function read(
    ErrorCode&BackedEnum $error
): ResolvedErrorDefinition
```

Method tersebut tidak mengembalikan object `ErrorDefinition` secara langsung.

## Alternatif: Mengembalikan `ErrorDefinition`

Reader memang membaca object `ErrorDefinition`, tetapi object tersebut hanya berisi metadata.

```php
#[ErrorDefinition(
    message: 'Draft sedang dikunci.',
    category: ErrorCategory::BUSINESS,
    httpStatus: 409,
)]
```

Kode error berada pada enum:

```php
case DRAFT_LOCKED = 'SM-WF-001';
```

Apabila Reader hanya mengembalikan `ErrorDefinition`, consumer harus membawa enum secara terpisah untuk memperoleh kode error.

```php
$error->value;
$definition->message;
$definition->category;
```

Pendekatan tersebut tidak dipilih karena membagi informasi yang secara konseptual merupakan satu Error Definition ke dalam dua object.

## `ResolvedErrorDefinition`

Reader menggabungkan informasi dari enum dan attribute menjadi satu object.

```text
Backed Enum
        +
ErrorDefinition Attribute
        │
        ▼
ResolvedErrorDefinition
```

Consumer cukup menggunakan satu object.

```php
$error = $reader->read(
    SuratMasukError::DRAFT_LOCKED
);

$error->code;
$error->message;
$error->category;
$error->severity;
$error->httpStatus;
```

Consumer tidak perlu mengetahui sumber setiap properti tersebut.

## Alasan

Pendekatan ini menyediakan API yang lebih sederhana. Enum dan attribute tidak perlu dibawa secara bersamaan, dan object hasil resolve dapat langsung digunakan untuk JSON response, exception, generator, logging, maupun validation.

Penambahan metadata baru juga tidak memerlukan perubahan pada consumer selama metadata tersebut dipetakan oleh Reader dan tersedia pada DTO.

## Konsekuensi

- Seluruh consumer menggunakan satu object hasil resolve.
- Tidak terjadi duplikasi pembacaan enum pada consumer.
- API Reader lebih sederhana.
- Object hasil resolve lebih mudah diserialisasi.

Keputusan ini menjadi dasar bagi `ApplicationException`, FormRequest Integration, JSON Response, dan Error Catalog Generator.

# 1.3.3 Input

## Keputusan

Method `read()` menerima enum yang mengimplementasikan marker interface `ErrorCode` dan merupakan `BackedEnum`.

```php
public function read(
    ErrorCode&BackedEnum $error
): ResolvedErrorDefinition
```

Kontrak tersebut memastikan Reader hanya menerima enum yang didefinisikan sebagai error code.

## Marker Interface `ErrorCode`

PHP menyediakan interface `BackedEnum`, tetapi interface tersebut terlalu umum. Enum berikut sama-sama merupakan `BackedEnum` meskipun bukan Error Definition.

```php
enum UserStatus: string
{
    case ACTIVE = 'active';
    case INACTIVE = 'inactive';
}

enum DocumentType: string
{
    case PDF = 'pdf';
}
```

Reader harus menolak enum yang bukan error. Oleh karena itu framework menyediakan marker interface khusus.

```php
interface ErrorCode
{
}
```

Enum error mengimplementasikan interface tersebut.

```php
enum SuratMasukError: string implements ErrorCode
{
    #[ErrorDefinition(...)]
    case DRAFT_LOCKED = 'SM-WF-001';
}
```

## Alternatif: Method pada Interface

Interface dapat mendefinisikan method untuk mengambil kode error.

```php
interface ErrorCode
{
    public function code(): string;
}
```

Pendekatan tersebut tidak dipilih. Seluruh enum yang diterima Reader telah merupakan `BackedEnum`, sehingga kode error dapat diperoleh langsung melalui `$error->value`.

Method `$error->code()` hanya akan menduplikasi mekanisme yang telah disediakan oleh PHP.

## Alternatif: Menggunakan `BackedEnum`

```php
public function read(
    BackedEnum $error
)
```

Pendekatan ini menerima semua backed enum, termasuk `Status::ACTIVE`, `Role::ADMIN`, dan `Color::RED`. Kontrak tersebut tidak menjamin bahwa enum yang diterima merupakan Error Definition.

## Alternatif: Menggunakan `string`

```php
public function read(string $code)
```

Pendekatan ini menghilangkan manfaat enum. Reader tidak lagi dapat mengetahui enum asal, membaca attribute, atau melakukan type checking terhadap input.

## Alasan

Kombinasi `ErrorCode` dan `BackedEnum` memberikan kontrak yang type-safe, self-documenting, dan tidak bergantung pada struktur folder. Consumer cukup memberikan enum error yang akan dibaca.

## Konsekuensi

- Hanya Error Definition yang dapat diproses.
- Enum lain ditolak oleh type system.
- Kontrak Reader lebih jelas.
- API tetap sederhana.

# 1.3.4 MissingErrorDefinitionException

## Keputusan

Apabila enum yang mengimplementasikan `ErrorCode` tidak memiliki attribute `ErrorDefinition`, Reader melempar `MissingErrorDefinitionException`.

```php
throw new MissingErrorDefinitionException(...);
```

Reader tidak mengembalikan `null`, `false`, atau object kosong.

## Alasan

Implementasi `ErrorCode` menyatakan bahwa enum tersebut merupakan Error Definition yang valid. Ketiadaan metadata merupakan kesalahan konfigurasi programmer, bukan business error.

Framework menerapkan pendekatan **fail fast** agar kesalahan konfigurasi diketahui pada titik pertama kali ditemukan.

## Alternatif: Mengembalikan `null`

```php
public function read(...): ?ResolvedErrorDefinition
```

Consumer kemudian harus melakukan pengecekan:

```php
if ($definition === null) {
    ...
}
```

Pendekatan ini tidak dipilih karena metadata yang hilang bukan kondisi normal. Jika Reader mengembalikan `null`, setiap consumer—termasuk ApplicationException, Validation, Generator, Logging, dan Response Builder—harus menangani kondisi yang seharusnya tidak terjadi.

Akibatnya, kompleksitas berpindah dari Reader ke seluruh consumer. Framework memilih menghentikan proses pada Reader.

## Alternatif: Exception Umum

Exception berikut dapat digunakan:

```php
RuntimeException
LogicException
Exception
```

Pendekatan tersebut tidak dipilih karena tidak menjelaskan penyebab kegagalan secara spesifik.

`MissingErrorDefinitionException` memberikan nama yang langsung menjelaskan penyebab error, dapat ditangkap secara spesifik, dan memungkinkan jenis kegagalan lain dibedakan pada masa mendatang.

## Konsekuensi

- Kontrak Reader tetap sederhana.
- Consumer tidak perlu melakukan null checking.
- Kesalahan konfigurasi langsung diketahui.
- Debugging lebih mudah.
- Framework mengikuti prinsip fail fast.

# 1.3.5 Reflection & In-Memory Cache

## Keputusan

`ErrorDefinitionReader` menggunakan PHP Reflection untuk membaca metadata error dan menyimpan hasil resolve pada cache internal berbasis memory.

```php
final class ErrorDefinitionReader
{
    /**
     * @var array<string, ResolvedErrorDefinition>
     */
    private array $cache = [];
}
```

Reflection hanya dilakukan saat metadata pertama kali dibaca. Pemanggilan berikutnya mengembalikan object yang tersimpan di cache.

## Alur Kerja

```text
Consumer
    │
    ▼
Reader
    │
    ├── Cache Hit
    │       │
    │       ▼
    │   Return Result
    │
    └── Cache Miss
            │
            ▼
      PHP Reflection
            │
            ▼
   Read ErrorDefinition
            │
            ▼
Build ResolvedErrorDefinition
            │
            ▼
 Store In Cache
            │
            ▼
Return Result
```

Reflection hanya dijalankan pada **cache miss**.

## Alasan Menggunakan Reflection

Metadata error didefinisikan menggunakan PHP Attribute.

```php
#[ErrorDefinition(
    message: 'Draft sedang dikunci.',
    category: ErrorCategory::BUSINESS,
    httpStatus: 409,
)]
```

Reflection merupakan mekanisme standar PHP untuk membaca metadata tersebut. Framework menggunakan kemampuan native PHP dan tidak memerlukan mekanisme registrasi manual.

## Alasan Menggunakan Cache

Reflection memiliki biaya lebih tinggi dibandingkan akses object biasa. Metadata error bersifat immutable sehingga hasil pembacaan tidak berubah selama lifecycle aplikasi.

Cache mencegah Reflection dan proses resolve dilakukan berulang kali. Setelah cache terisi, pemanggilan berikutnya cukup melakukan lookup pada array.

## Data yang Disimpan pada Cache

### Alternatif: Menyimpan Reflection

```text
ReflectionEnum
ReflectionAttribute
ReflectionClass
```

Pendekatan ini tidak dipilih karena consumer tetap memerlukan proses resolve setiap kali object diambil. Reader masih harus membaca attribute, membuat object baru, dan menggabungkan metadata.

### Alternatif: Menyimpan `ErrorDefinition`

```text
Cache

↓

ErrorDefinition
```

Pendekatan ini juga tidak dipilih karena consumer masih membutuhkan error code dan Reader tetap harus membangun object baru setiap kali dipanggil.

### Keputusan: Menyimpan `ResolvedErrorDefinition`

```text
Cache

↓

ResolvedErrorDefinition
```

Object yang telah selesai di-resolve disimpan secara langsung. Dengan demikian, Reflection dan proses resolve hanya dilakukan satu kali, dan consumer langsung memperoleh object yang siap digunakan.

## Alternatif: Laravel Cache

```php
Cache::remember(...)
```

Pendekatan ini tidak dipilih karena metadata Error Definition bersifat immutable, berukuran kecil, dan hanya diperlukan di dalam process. Laravel Cache menambah dependency dan kompleksitas tanpa manfaat yang sebanding.

Framework menggunakan cache internal sederhana.

```php
private array $cache = [];
```

## Kompatibilitas Laravel Octane

Pendekatan ini aman digunakan pada Laravel Octane selama cache hanya menyimpan data immutable, seperti:

- code;
- message;
- category;
- severity;
- retryable;
- httpStatus.

Reader tidak boleh menyimpan data yang dapat berubah antar request, seperti:

- Request;
- User;
- Locale;
- Tenant;
- Service;
- Exception;
- Runtime Context.

### Konsekuensi pada Long-Running Process

Pada worker Laravel Octane atau Queue Worker, perubahan source code tidak langsung terlihat karena PHP memuat class saat process dimulai. Worker harus di-restart setelah source code berubah. Kondisi ini bukan disebabkan oleh cache Reader.

## Konsekuensi

- Reflection hanya dilakukan satu kali untuk setiap metadata yang dibaca.
- Proses resolve hanya dilakukan satu kali.
- Consumer memperoleh object siap pakai.
- Reader tidak bergantung pada Laravel Cache.
- Reader aman digunakan pada Laravel Octane selama hanya menyimpan data immutable.
- Cache hanya hidup selama lifecycle Reader.
- Implementasi Reader direkomendasikan menggunakan singleton agar cache tidak dibuat ulang setiap kali Reader dibuat.
- Setiap Error Definition yang pernah dibaca tetap berada di memory selama lifecycle tersebut.

# 1.3.6 Resolved DTO Tanpa Enum Source

## Keputusan

`ResolvedErrorDefinition` hanya berisi hasil resolve. Object ini tidak menyimpan enum source, Reflection, `ReflectionAttribute`, `ReflectionEnum`, atau object Reflection lainnya.

## Alternatif: Menyimpan Enum Source

```php
final readonly class ResolvedErrorDefinition
{
    public ErrorCode $source;
}
```

Pendekatan ini tidak dipilih karena consumer tidak lagi memerlukan enum setelah Reader menyelesaikan proses resolve. Consumer hanya membutuhkan:

- code;
- message;
- category;
- severity;
- retryable;
- httpStatus.

Menyimpan enum hanya menambah ukuran object tanpa memberikan informasi yang diperlukan oleh consumer.

## Alternatif: Menyimpan Reflection

Reflection merupakan detail implementasi Reader. Consumer tidak perlu mengetahui bahwa metadata berasal dari Reflection.

Framework memisahkan hasil resolve dari mekanisme pembacaan metadata. Karena itu, Reflection tidak disimpan pada DTO.

## Alasan

Reader bertugas mengubah metadata menjadi object siap pakai, bukan mengembalikan object Reflection. Seluruh consumer dapat bekerja menggunakan satu DTO tanpa bergantung pada mekanisme pembacaan maupun enum source.

## Konsekuensi

- DTO tetap sederhana.
- DTO mudah diserialisasi.
- DTO mudah diuji.
- DTO tidak bergantung pada Reflection.
- DTO tidak bergantung pada enum source.
