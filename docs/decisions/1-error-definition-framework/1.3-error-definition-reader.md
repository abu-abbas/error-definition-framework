# 1.3 ErrorDefinitionReader

[← Kembali ke 1. Error Definition Framework](./README.md)

> **TL;DR:** Reader mengambil satu definisi error dari source code dan menyiapkannya menjadi data lengkap yang dapat dipakai ulang oleh exception, validation, response, logging, dan generator.

## Tujuan

Menetapkan tanggung jawab, input, output, dan perilaku kegagalan `ErrorDefinitionReader`.

## Keputusan

### Tanggung Jawab

Reader hanya membaca satu enum case, mengambil attribute `ErrorDefinition`, melakukan validasi dasar, dan mengembalikan metadata yang telah di-resolve.

Reader tidak melakukan discovery, pemindaian folder, validasi duplicate error code, pembuatan catalog, logging, registry, knowledge base, monitoring, atau integrasi ITSM.

### Output

Method `read()` mengembalikan `ResolvedErrorDefinition`, bukan `ErrorDefinition` mentah.

### Input

Reader menerima enum yang mengimplementasikan marker interface `ErrorCode` dan merupakan `BackedEnum`.

### Metadata yang Tidak Ditemukan

Enum `ErrorCode` yang tidak memiliki attribute `ErrorDefinition` menghasilkan `MissingErrorDefinitionException`. Reader tidak mengembalikan `null`, `false`, atau object kosong.

### Bentuk DTO

`ResolvedErrorDefinition` hanya menyimpan hasil resolve. DTO tidak menyimpan enum source maupun object Reflection.

## Alasan

### Reading Dipisahkan dari Discovery

Reading menerima satu enum case dan membaca definisinya. Discovery menemukan seluruh Error Definition dalam aplikasi. Keduanya memiliki consumer dan alasan perubahan yang berbeda.

Alternatif berupa method `discover()` atau `scan()` pada Reader tidak dipilih karena mencampur tanggung jawab dan membuat Reader bergantung pada struktur project.

### Output Harus Membawa Error Code

`ErrorDefinition` hanya berisi metadata dari attribute, sedangkan error code berada pada enum. Mengembalikan `ErrorDefinition` secara langsung mengharuskan consumer membawa dua object.

`ResolvedErrorDefinition` menggabungkan keduanya sehingga consumer memperoleh satu object yang siap digunakan dan diserialisasi.

### Input Harus Spesifik terhadap Error Code

`BackedEnum` saja terlalu umum karena juga mencakup enum status, role, atau tipe dokumen. Marker interface `ErrorCode` menyatakan bahwa enum memang dimaksudkan sebagai error code.

Method tambahan seperti `code(): string` pada interface tidak dipilih karena kode sudah tersedia melalui value Backed Enum. Input berupa string juga tidak dipilih karena menghilangkan informasi enum asal, pembacaan attribute, dan type checking.

### Konfigurasi yang Tidak Valid Harus Fail Fast

Implementasi `ErrorCode` menyatakan bahwa enum merupakan Error Definition yang valid. Ketiadaan attribute adalah kesalahan konfigurasi programmer, bukan kondisi bisnis normal.

Return `null` akan memindahkan null checking ke seluruh consumer. Exception umum juga tidak dipilih karena tidak menjelaskan penyebab kegagalan secara spesifik.

### DTO Tidak Membocorkan Detail Implementasi

Setelah proses resolve selesai, consumer hanya membutuhkan code, message, category, severity, retryable, dan HTTP status. Enum serta Reflection merupakan detail implementasi Reader.

Mengeluarkan detail tersebut dari DTO membuat hasil resolve mudah diserialisasi, diuji, dan digunakan tanpa ketergantungan pada mekanisme pembacaannya.

## Konsekuensi

- Reader tetap kecil, terfokus, dan mudah diuji.
- Consumer tidak perlu mengetahui lokasi enum atau mekanisme Reflection.
- Hanya enum error yang dapat diproses; enum lain ditolak oleh type system.
- Kesalahan konfigurasi langsung menghasilkan exception yang spesifik.
- Discovery dapat berkembang tanpa mengubah kontrak Reader.
- Implementasi dapat dimulai tanpa mengubah kontrak publik yang telah disepakati.
