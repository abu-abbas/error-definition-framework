# 1.3.1 Caching dan Siklus Proses

[← Kembali ke 1. Error Definition Framework](./README.md)

> **TL;DR:** metadata error dibaca sekali dan disimpan selama process berjalan, tetapi cache hanya boleh berisi data tetap agar aman pada worker yang melayani banyak request.

## Tujuan

Menetapkan mekanisme pembacaan metadata, bentuk cache, dan batas keamanan Reader pada PHP process biasa maupun long-running process.

## Keputusan

Reader menggunakan PHP Reflection untuk membaca attribute `ErrorDefinition`.

Hasil resolve disimpan sebagai `ResolvedErrorDefinition` pada in-memory cache internal selama lifecycle Reader. Reflection dan proses resolve hanya dilakukan ketika cache belum memiliki hasil untuk enum case tersebut.

Reader direkomendasikan terdaftar sebagai singleton agar instance dan cache digunakan ulang selama lifecycle process.

Cache hanya boleh menyimpan metadata immutable. Reader tidak boleh menyimpan Request, User, Locale, Tenant, Service, Exception, atau Runtime Context.

## Alasan

Reflection merupakan mekanisme native PHP untuk membaca Attribute. Metadata Error Definition tidak berubah selama source code yang sama sedang dijalankan, sehingga pembacaan dan resolve berulang tidak memberikan nilai tambahan.

Cache menyimpan hasil akhir, bukan Reflection atau `ErrorDefinition` mentah. Menyimpan bentuk mentah tetap mengharuskan Reader membangun DTO pada setiap pemanggilan.

Laravel Cache tidak dipilih karena metadata berukuran kecil, immutable, dan hanya diperlukan di dalam process. Cache internal memenuhi kebutuhan tanpa menambah dependency, serialization, atau storage eksternal.

✅ **Contoh:** code, message, category, severity, retryable, dan HTTP status boleh disimpan karena nilainya tetap selama process berjalan.

❌ **Bukan contoh:** User atau Locale aktif tidak boleh disimpan karena nilainya dapat berbeda pada request berikutnya yang dilayani process yang sama.

Pada Laravel Octane dan Queue Worker, perubahan source code tetap membutuhkan restart worker. Kondisi tersebut merupakan karakteristik long-running process, bukan kegagalan cache Reader.

## Konsekuensi

- Reflection dan resolve hanya dilakukan saat cache miss.
- Cache menggunakan sedikit memory untuk setiap Error Definition yang pernah dibaca.
- Membuat Reader berulang kali akan membuat cache berulang; binding singleton menjadi konfigurasi yang direkomendasikan.
- Reader aman digunakan pada Octane dan Queue Worker selama cache hanya berisi data immutable.
- Worker harus di-restart setelah perubahan source code agar definisi baru dimuat.
