# 1.4 ApplicationException

[← Kembali ke 1. Error Definition Framework](./README.md)

> **TL;DR:** `ApplicationException` membawa definisi error yang sudah di-resolve, context internal, dan exception penyebab tanpa menangani logging atau response HTTP.

## Tujuan

Menetapkan tanggung jawab dan kontrak `ApplicationException` sebagai exception standar aplikasi.

## Keputusan

`ApplicationException` merupakan `final class` yang extends `RuntimeException` dan hanya berfungsi sebagai data carrier.

Constructor menerima:

- `ResolvedErrorDefinition`;
- runtime context berupa `array<string, mixed>`;
- optional previous `Throwable`.

Definition dan context diekspos sebagai public readonly properties. Message selalu berasal dari `ResolvedErrorDefinition` dan tidak dapat di-override. Property bawaan `Exception::$code` tetap bernilai `0`; string error code diakses melalui definition.

Runtime context hanya digunakan untuk logging dan debugging. Context tidak otomatis dikirim ke client dan harus disanitasi pada logging atau reporting layer sebelum diteruskan ke log, Sentry, atau monitoring.

`ApplicationException` tidak melakukan resolve enum, render response HTTP, logging, maupun sanitasi context.

## Alasan

Menerima `ResolvedErrorDefinition` membuat dependency eksplisit dan memastikan exception langsung membawa data yang siap digunakan. Menerima enum secara langsung akan memaksa exception melakukan proses resolve, sedangkan factory akan menambah abstraction yang belum diperlukan.

Message tidak dapat di-override agar Error Definition tetap menjadi satu sumber kebenaran. Detail kejadian ditempatkan pada runtime context tanpa mengubah pesan yang ditampilkan kepada pengguna.

Array dipilih untuk context karena bentuk data logging berbeda pada setiap kejadian. Context DTO dapat ditambahkan kemudian jika ditemukan struktur stabil yang digunakan berulang kali.

Previous exception dipertahankan agar penyebab awal dan stack trace tidak hilang ketika exception lain dibungkus sebagai `ApplicationException`.

Public readonly properties dipilih karena class ini tidak memiliki behavior selain membawa data. Getter tambahan tidak memberikan nilai pada kontrak saat ini.

Class bersifat final agar variasi error tetap direpresentasikan melalui enum dan metadata, bukan melalui hierarchy exception per modul.

`RuntimeException` dipilih karena exception merepresentasikan kegagalan ketika aplikasi menjalankan operasi. Menggunakan base `Exception` tidak memberikan klasifikasi tambahan.

String error code tidak dipaksakan masuk ke `Exception::$code` karena property tersebut menerima integer. Konversi atau mapping integer akan menambah source of truth dan berisiko kehilangan makna kode.

Sanitasi context ditempatkan pada logging boundary agar aturan keamanan diterapkan terpusat. Masking manual oleh setiap caller mudah terlupa, sedangkan sanitasi di dalam exception akan mencampur data carrier dengan kebijakan logging.

## Konsekuensi

- Caller harus melakukan resolve Error Definition sebelum membuat exception.
- Seluruh message aplikasi tetap berasal dari Error Definition.
- Context dapat digunakan untuk observability tanpa menjadi bagian dari response publik.
- Logging dan reporting wajib menyanitasi context sebelum mengirimkannya ke sistem lain.
- Exception chain tetap tersedia melalui previous `Throwable`.
- Perbedaan error tidak menghasilkan subclass exception baru.
- Renderer dan logger dapat berkembang tanpa mengubah kontrak `ApplicationException`.
