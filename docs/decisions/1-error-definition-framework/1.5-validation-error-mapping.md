# 1.5 Validation Error Mapping

[← Kembali ke 1. Error Definition Framework](./README.md)

> **TL;DR:** FormRequest tetap bekerja seperti Laravel biasa, tetapi setiap validation failure dipetakan ke Error Definition yang type-safe dan siap digunakan oleh response standar.

## Tujuan

Menetapkan cara FormRequest menghubungkan validation rule Laravel dengan Error Definition tanpa mengganti lifecycle validation Laravel.

## Keputusan

FormRequest melakukan opt-in melalui trait `HasErrorDefinitions` dan menyediakan method `errorCodes()` dengan mapping `attribute.rule` ke enum error.

Request yang menggunakan trait wajib memetakan seluruh rule yang gagal. Mapping yang tidak ditemukan menghasilkan `MissingValidationErrorCodeException`.

Nested attribute menggunakan dot notation dan wildcard Laravel. Exact match diperiksa lebih dulu, kemudian wildcard. Built-in rule menggunakan nama Laravel, sedangkan custom Rule object menggunakan FQCN sebagai identifier.

Framework memproses seluruh failure yang diberikan Laravel dan tidak mengubah behavior `bail`, stop-on-first-failure, urutan rule, redirect, maupun named error bag.

Message berasal dari Error Definition dan dapat menggunakan placeholder Laravel seperti `:attribute`, `:min`, dan `:max`. FormRequest yang menggunakan trait tidak boleh mendefinisikan `messages()` sendiri, tetapi tetap boleh menggunakan `attributes()`.

Validation tambahan melalui `after()` menggunakan helper `addValidationError()` agar error manual tetap memiliki attribute, rule identifier, dan Error Definition.

Validation failure menghasilkan `ErrorValidationException` yang extends `ValidationException` dan membawa list `ResolvedValidationError`.

Error Definition yang digunakan oleh validation wajib memiliki category `VALIDATION`, HTTP status `422`, dan `retryable: false`.

## Alasan

Method `errorCodes()` mengikuti pola `messages()` Laravel sehingga mapping tetap familiar tanpa membungkus seluruh built-in rule dengan object baru. Trait dipilih agar aplikasi tetap menggunakan FormRequest biasa dan tidak diwajibkan mewarisi base request khusus.

Strict mapping menjaga seluruh validation failure tetap memiliki structured error code. FormRequest yang belum dimigrasikan dapat tetap menggunakan Laravel native tanpa trait.

Pemeriksaan mapping dilakukan ketika rule benar-benar gagal. Eager parsing seluruh `rules()` tidak dipilih karena conditional rule, closure, wildcard, `sometimes`, custom rule, dan fluent rule builder membuat pemeriksaan runtime menjadi kompleks dan rentan false positive. Audit menyeluruh menjadi tugas Error Linter pada Stage 1.8.

Dot notation dan wildcard mengikuti cara Laravel merepresentasikan nested input. Exact-only mapping tidak dapat menangani index array dinamis, sedangkan mapping berdasarkan rule saja kehilangan konteks field dan modul.

Custom Rule object menggunakan FQCN untuk mencegah collision antar-class bernama sama. Error code tetap berada pada FormRequest agar custom rule dapat digunakan ulang oleh modul berbeda.

Placeholder diproses oleh Laravel agar nilai rule dan message tetap sinkron. Message literal dapat menjadi usang ketika parameter rule berubah, sedangkan memakai message Laravel akan menghilangkan Error Definition sebagai source of truth.

`ErrorValidationException` tetap merupakan `ValidationException`, sehingga integrasi Laravel untuk redirect, error bag, dan JSON detection tetap berlaku. `ApplicationException` tidak dipilih karena satu validation request dapat menghasilkan banyak error.

`ResolvedValidationError` menggunakan DTO karena bentuk datanya stabil dan menjadi kontrak antara validation exception dan renderer. Array tidak dipilih karena key mudah typo dan tidak type-safe.

✅ **Contoh:** `attachments.2.file.required` dicocokkan dengan `attachments.*.file.required`, karena index merupakan bagian dinamis dari input array.

❌ **Bukan contoh:** semua kegagalan `required` dipetakan ke satu error global, karena field dan konteks bisnisnya menjadi tidak dapat dibedakan.

## Konsekuensi

- Request yang opt-in harus memetakan setiap rule yang dapat gagal.
- Kesalahan mapping menjadi configuration error dan gagal secara eksplisit.
- Message, code, category, severity, dan status berasal dari Error Definition yang sama.
- FormRequest tetap dapat memakai fitur native Laravel seperti wildcard, custom rule, `after()`, `bail`, attributes, redirect, dan error bag.
- Business rule dengan category atau status selain validation harus dilempar sebagai `ApplicationException` dari application atau service layer.
- Renderer menerima structured validation errors tanpa menghitung ulang mapping.
- Linter diperlukan untuk menemukan mapping yang belum pernah mencapai jalur gagal saat testing.
