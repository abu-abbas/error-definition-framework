# 1.6 Format Response JSON Standar

[← Kembali ke 1. Error Definition Framework](./README.md)

> **TL;DR:** response error mempertahankan bentuk yang familiar bagi pengguna Laravel, tetapi menambahkan error code terstruktur yang dapat digunakan client tanpa membaca message.

## Tujuan

Menetapkan bentuk response JSON untuk `ApplicationException` dan `ErrorValidationException`, metadata yang boleh dikirim kepada client, serta tanggung jawab komponen yang membentuk response.

## Keputusan

### Application Error

`ApplicationException` menghasilkan response berikut:

```json
{
  "message": "Draft sedang dikunci.",
  "code": "SM-WF-001",
  "retryable": false
}
```

Nilai `message`, `code`, dan `retryable` berasal dari `ResolvedErrorDefinition`. HTTP status menggunakan `definition.httpStatus` dan tidak diduplikasi pada body JSON.

Category dan severity tetap menjadi metadata internal untuk logging, reporting, dan registry. Runtime context, exception class, file, stack trace, serta previous exception tidak dikirim kepada client, termasuk ketika debug mode aktif.

### Validation Error

`ErrorValidationException` menghasilkan response yang dikelompokkan berdasarkan attribute aktual:

```json
{
  "message": "Nomor surat wajib diisi. (dan 1 error lainnya)",
  "errors": {
    "nomor_surat": [
      {
        "code": "SM-VAL-001",
        "message": "Nomor surat wajib diisi.",
        "retryable": false
      }
    ],
    "attachments.2.file": [
      {
        "code": "SM-VAL-002",
        "message": "File lampiran wajib diisi.",
        "retryable": false
      }
    ]
  }
}
```

Top-level `message` menggunakan summary dari `ValidationException` Laravel. Setiap validation error membawa public fields yang sama dengan application error, sedangkan nama rule tetap menjadi informasi internal.

Wildcard hanya digunakan ketika mencari mapping. Key pada response selalu menggunakan attribute aktual, misalnya `attachments.2.file`, agar client dapat menghubungkan error ke input yang tepat.

Validation response menggunakan HTTP status `422`. Response tidak memiliki kode error buatan pada level atas karena satu request dapat menghasilkan beberapa Error Definition yang berbeda.

### Rendering

Response dibentuk oleh central `ErrorResponseRenderer`. Exception tetap berfungsi sebagai data carrier dan tidak memiliki method `render()`.

Laravel Service Provider mendaftarkan renderer secara otomatis untuk `ApplicationException` dan `ErrorValidationException`. Renderer hanya digunakan ketika mekanisme native Laravel menentukan bahwa request mengharapkan JSON. Request non-JSON dan exception di luar framework tetap ditangani oleh Laravel.

## Alasan

Format tanpa object pembungkus `error` mempertahankan pola response Laravel dan mengurangi lapisan yang tidak diperlukan. Client dapat menampilkan `message`, menggunakan `code` untuk behavior spesifik, dan membaca `retryable` untuk menentukan apakah operasi dapat dicoba kembali.

Category dan severity tidak dikirim karena keduanya merupakan metadata operasional. Membatasi kontrak publik pada data yang digunakan client mengurangi ketergantungan dan memungkinkan kebijakan internal berkembang tanpa mengubah API.

Validation error dikelompokkan berdasarkan field agar mudah digunakan oleh form. Flat list tidak dipilih karena client harus mengelompokkan ulang error sebelum menampilkannya pada input yang sesuai.

Rule validation tidak dikirim karena merupakan detail implementasi Laravel. Error code menjadi identifier publik yang stabil meskipun rule atau implementasi backend berubah.

Summary native Laravel dipertahankan agar localization dan perilaku `ValidationException` tidak dibuat ulang oleh framework.

Central renderer menjaga response policy berada di satu tempat. Method `render()` pada exception tidak dipilih karena akan mencampurkan data kegagalan dengan kebijakan pembentukan response HTTP.

Registrasi otomatis menghindari integration code yang berbeda pada setiap aplikasi. Scope tetap terbatas karena renderer hanya menangani exception framework dan mengikuti deteksi JSON Laravel.

Debug fields tidak pernah ditambahkan agar kontrak response tetap sama pada setiap environment dan informasi internal tidak terekspos akibat kesalahan konfigurasi.

Normalisasi seluruh exception Laravel tidak menjadi bagian keputusan ini. Melakukannya membutuhkan Error Definition dan mapping eksplisit untuk setiap jenis kegagalan yang belum dirancang.

## Konsekuensi

- `message`, `code`, dan `retryable` menjadi kontrak publik application error.
- Validation response tetap dikelompokkan per field dan setiap failure mempertahankan Error Definition masing-masing.
- Client harus menggunakan `code`, bukan `message` atau nama rule, untuk behavior programatik.
- Category, severity, dan runtime context tetap tersedia bagi komponen internal tanpa menjadi bagian dari API publik.
- Perubahan nama atau bentuk field response merupakan breaking change.
- Aplikasi memperoleh renderer setelah Service Provider aktif tanpa registrasi manual.
- Exception non-framework tetap mengikuti response handling Laravel.
