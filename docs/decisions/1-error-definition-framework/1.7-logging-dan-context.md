# 1.7 Logging dan Context

[← Kembali ke 1. Error Definition Framework](./README.md)

> **TL;DR:** setiap application error dicatat satu kali dengan metadata terstruktur, sedangkan context kejadian dibersihkan dari data sensitif sebelum masuk log.

## Tujuan

Menetapkan cara `ApplicationException` dilaporkan, pemetaan severity ke log level, struktur log context, dan batas sanitasi data sensitif.

## Keputusan

### Reporting

`ApplicationExceptionReporter` menangani reporting melalui pipeline exception Laravel. `ApplicationException` dan `ErrorResponseRenderer` tidak melakukan logging.

Setiap `ApplicationException` yang mencapai exception handler dicatat tepat satu kali. Reporter menghentikan default reporting Laravel untuk exception tersebut agar event yang sama tidak tercatat dua kali.

`ErrorValidationException` tidak dicatat oleh reporter. Validation failure merupakan input pengguna yang dapat diperkirakan dan akan menghasilkan volume log yang tinggi tanpa memberi nilai operasional yang sebanding.

Log level ditentukan dari severity:

| Severity | PSR-3 Log Level |
|----------|-----------------|
| Low | `info` |
| Medium | `warning` |
| High | `error` |
| Critical | `critical` |

Message log berasal dari Error Definition. Metadata framework ditempatkan pada root log context, sedangkan runtime context dari caller ditempatkan di dalam key `context`:

```php
[
    'error_code' => 'SM-WF-001',
    'category' => 'workflow',
    'severity' => 'medium',
    'retryable' => false,
    'http_status' => 409,
    'context' => [
        'user_id' => 42,
        'document_id' => 1001,
    ],
    'exception' => $exception,
]
```

Exception disimpan pada PSR-3 key `exception` agar stack trace dan previous exception tetap tersedia. Runtime context tidak digabungkan ke root karena key milik caller dapat bertabrakan dengan metadata framework.

Reporter menggunakan default Laravel log channel melalui PSR-3. Package tidak membuat channel khusus dan tidak bergantung langsung pada Sentry atau monitoring provider lain. Aplikasi dapat meneruskan log melalui konfigurasi logging Laravel yang sudah dimilikinya.

### Context Boundary

Reporter hanya mencatat context yang diberikan secara eksplisit ketika `ApplicationException` dibuat. Package tidak mengambil request body, headers, session, user, atau correlation ID secara otomatis.

Middleware atau caller dapat memasukkan correlation ID dan identifier lain ke runtime context apabila tersedia. Package tidak menghasilkan correlation ID sendiri.

### Sanitasi

Seluruh runtime context melewati `ContextSanitizer` sebelum diteruskan ke logger. Sanitasi berjalan secara recursive pada nested array dan mengganti value sensitif dengan `[REDACTED]` tanpa mengubah key atau struktur context.

Sensitive key bawaan disimpan di dalam package dan tidak dapat dihapus oleh konfigurasi aplikasi:

```text
password
password_confirmation
current_password
token
access_token
refresh_token
authorization
cookie
secret
api_key
client_secret
private_key
```

Aplikasi dapat menambahkan key khusus domain melalui `logging.additional_sensitive_keys` pada `config/error-definition.php`. Daftar tambahan digabungkan dengan baseline package, bukan menggantikannya.

Pencocokan key dilakukan dengan mengubah huruf menjadi lower-case dan menghapus karakter selain huruf dan angka. Dengan demikian, `access_token`, `access-token`, `accessToken`, dan `ACCESS_TOKEN` dianggap sama. Pencocokan menggunakan nama key lengkap, bukan substring.

Scalar dan `null` dipertahankan. Nested array diproses secara recursive, `BackedEnum` diubah menjadi value, dan `DateTimeInterface` diubah menjadi ISO 8601. Arbitrary object dan resource tidak diserialisasi; sanitizer hanya mencatat tipenya.

Sanitizer tidak memindai pola isi value. Deteksi berdasarkan value tidak dipilih karena mudah menghasilkan false positive dan tidak dapat menentukan sensitivitas data domain secara andal.

Object `ApplicationException` pada key PSR-3 `exception` tidak diproses oleh context sanitizer. Exception message tidak boleh memuat credential atau data sensitif karena stack trace dan previous exception diperlukan untuk diagnosis.

## Alasan

Central reporter menjaga kebijakan logging berada pada satu boundary. Logging di dalam exception akan mencampurkan data carrier dengan side effect, sedangkan logging di renderer hanya mencakup exception yang menghasilkan response JSON.

Semua `ApplicationException` dicatat karena severity sudah menyediakan cara membedakan tingkat kepentingan event. Membatasi reporting hanya pada High dan Critical akan menghilangkan jejak operasional Low dan Medium yang tetap berguna ketika menganalisis masalah.

Pemetaan severity menghasilkan log level yang konsisten tanpa meminta setiap caller memilih level sendiri. Menggunakan satu level `error` untuk seluruh kejadian akan membuat business error berdampak rendah terlihat sama dengan gangguan layanan.

Context eksplisit lebih aman daripada mengambil seluruh request atau user secara otomatis. Caller mengetahui identifier yang diperlukan untuk menjelaskan kejadian, sedangkan automatic capture berisiko membawa credential, file, atau data pribadi yang tidak relevan.

Baseline sanitizer di dalam package memberikan perlindungan yang tidak dapat dilemahkan secara tidak sengaja. Config tambahan tetap diperlukan karena framework tidak dapat mengetahui seluruh nama data sensitif pada setiap domain aplikasi.

Normalisasi key menangani variasi gaya penamaan tanpa menggunakan substring matching yang terlalu luas. Sebagai contoh, key `token_count` tidak otomatis dianggap sama dengan `token`.

Arbitrary object tidak diserialisasi karena object dapat membawa relasi, state internal, atau property sensitif dalam jumlah besar. Identifier yang relevan harus dikirim secara eksplisit sebagai scalar pada runtime context.

PSR-3 dan default Laravel log channel membuat package tetap netral terhadap tujuan log. Integrasi monitoring dapat berkembang pada sisi aplikasi tanpa menambah dependency pada framework ini.

## Konsekuensi

- Setiap `ApplicationException` menghasilkan satu structured log event ketika mencapai exception handler.
- Validation failure tidak menambah noise pada application log.
- Caller bertanggung jawab memilih context minimum yang menjelaskan kejadian.
- Runtime context selalu disanitasi sebelum masuk logger atau monitoring melalui pipeline package.
- Default sensitive keys tetap terlindungi meskipun aplikasi memiliki konfigurasi tambahan.
- Data domain sensitif harus didaftarkan melalui `additional_sensitive_keys`.
- Arbitrary object tidak dapat digunakan untuk memasukkan seluruh model atau request ke log secara otomatis.
- Exception chain tetap tersedia dan log storage harus diperlakukan sebagai sistem internal yang terlindungi.
