# 1.8 Validator dan Linter

[← Kembali ke 1. Error Definition Framework](./README.md)

> **TL;DR:** kesalahan definisi dan mapping error diperiksa bersama sebelum deployment agar aplikasi tidak menunggu jalur gagal terjadi di production.

## Tujuan

Menetapkan tanggung jawab linter, target yang diperiksa, rule validasi, bentuk hasil, dan perilaku command untuk penggunaan lokal maupun CI.

## Keputusan

### Tanggung Jawab

`ErrorDefinitionLinter` memeriksa enum error dan FormRequest yang diberikan kepadanya. Linter tidak memindai folder, mencari class, atau menentukan struktur project.

Input linter berupa iterable enum class dan FormRequest class. [Discovery pada Decision 1.9](./1.9-error-definition-discovery-mechanism.md) menyediakan seluruh target aplikasi kepada linter tanpa membuat daftar manual di config.

Linter dijalankan secara eksplisit melalui test, command, atau CI. Pemeriksaan tidak berjalan ketika aplikasi boot dan tidak menjadi dependency runtime untuk menghasilkan response error.

### Hasil Pemeriksaan

Seluruh temuan dikumpulkan dalam `LintReport`; linter tidak berhenti pada error pertama. Setiap `LintFinding` memiliki:

- stable rule ID;
- level `error` atau `warning`;
- source berupa class, enum case, atau mapping key;
- message yang menjelaskan pelanggaran.

`error` digunakan untuk pelanggaran kontrak yang dapat dibuktikan. `warning` digunakan ketika struktur dinamis tidak dapat diaudit dengan pasti tanpa menebak perilaku aplikasi.

Linter hanya melaporkan masalah dan tidak mengubah source code. Autofix dan suppression tidak disediakan sampai terdapat kebutuhan nyata yang tidak dapat diselesaikan dengan memperbaiki definisi sumber.

### Rule Error Definition

Setiap target enum harus:

- merupakan enum yang mengimplementasikan `ErrorCode` dan `BackedEnum`;
- menggunakan string backing;
- memiliki tepat satu attribute `ErrorDefinition` pada setiap case;
- memiliki error code yang mengikuti format framework;
- memiliki message yang tidak kosong setelah whitespace dihapus;
- memiliki HTTP status antara `400` dan `599`.

Format error code menggunakan uppercase segments yang dipisahkan oleh `-` dan diakhiri serial minimal tiga digit:

```regex
^[A-Z0-9]+(?:-[A-Z0-9]+)+-\d{3,}$
```

```text
SM-WF-001       valid
IAM-AUTH-0001   valid
draft_locked    invalid
SM-01           invalid
```

Error code wajib unik di seluruh target yang diperiksa, bukan hanya di dalam satu enum.

Linter tidak menebak hubungan category, severity, retryability, dan HTTP status di luar kontrak yang telah ditetapkan. Sebagai contoh, category `NOT_FOUND` tidak otomatis dipaksa menggunakan status `404` karena framework belum menetapkan aturan universal tersebut.

### Rule Validation Mapping

Linter membuat FormRequest melalui Laravel container, lalu mengevaluasi `rules()` dan `errorCodes()`. Rule yang dikembalikan pada proses tersebut dibandingkan dengan mapping `attribute.rule`.

Untuk rule yang dapat diidentifikasi secara pasti:

- setiap rule wajib memiliki mapping;
- mapping key harus mengikuti bentuk `attribute.rule`;
- mapping value harus merupakan enum `ErrorCode` yang valid;
- mapping yang tidak memiliki pasangan rule menghasilkan warning;
- FormRequest yang opt-in tidak boleh meng-override `messages()`.

Error Definition yang digunakan oleh validation wajib memiliki category `VALIDATION`, HTTP status `422`, dan `retryable: false`.

Closure, conditional branch, atau rule object yang tidak menyediakan identifier stabil tidak ditebak oleh linter. Kondisi dinamis yang terlihat tetapi tidak dapat dipastikan menghasilkan warning. Branch yang tidak dikembalikan ketika `rules()` dievaluasi berada di luar jaminan audit dan tetap dilindungi oleh strict runtime mapping pada Decision 1.5.

Linter tidak menggunakan AST parser atau dependency static analysis tambahan. Pendekatan tersebut tidak dipilih karena kompleksitasnya tidak menjamin seluruh kondisi runtime Laravel dapat dibuktikan.

### Command dan Exit Code

Kontrak Artisan command adalah:

```shell
php artisan error-definition:lint
php artisan error-definition:lint --strict
```

Command menampilkan output human-readable dan summary. Format JSON belum disediakan.

- exit code `1` ketika terdapat error;
- exit code `0` ketika bersih atau hanya memiliki warning;
- opsi `--strict` membuat warning menghasilkan exit code `1`.

Command tidak melakukan folder scanning. Penyediaan target seluruh aplikasi menjadi tanggung jawab [discovery mechanism pada Decision 1.9](./1.9-error-definition-discovery-mechanism.md). Engine tetap dapat dipanggil langsung dengan daftar target eksplisit pada test atau tooling internal.

## Alasan

Memisahkan linting dari discovery menjaga engine dapat diuji dengan input deterministik dan tidak bergantung pada struktur folder. Daftar config manual tidak dipilih karena mudah tertinggal ketika enum atau FormRequest baru ditambahkan.

Menjalankan linter saat boot akan menambah biaya pada setiap process dan mengubah kesalahan development menjadi dependency runtime. Command dan CI memberi feedback sebelum deployment tanpa memengaruhi request aplikasi.

Mengumpulkan seluruh temuan mengurangi siklus perbaikan. Fail-fast cocok untuk Reader yang memproses satu enum case saat runtime, tetapi linter bertujuan memberi gambaran lengkap mengenai keadaan project.

Stable rule ID membuat hasil dapat diuji dan dicari tanpa bergantung pada perubahan wording message. Pemisahan error dan warning mencegah struktur Laravel yang dinamis menghasilkan kegagalan palsu.

Format kode yang konsisten membuat katalog dapat dibaca dan diurutkan lintas aplikasi. Pemeriksaan uniqueness global mencegah dua kondisi berbeda memiliki identifier yang sama.

Audit validation melalui Laravel container mengikuti cara FormRequest benar-benar dievaluasi tanpa membangun parser PHP sendiri. Batas audit dinyatakan secara eksplisit karena conditional rule tidak dapat dibuktikan hanya dari satu kondisi input.

Runtime strict mapping tetap diperlukan sebagai pertahanan terakhir. Linter menemukan kesalahan lebih awal, tetapi tidak menggantikan validasi ketika rule benar-benar gagal.

Autofix tidak dipilih karena linter tidak dapat menentukan message, category, severity, atau error code yang benar berdasarkan source code saja. Suppression juga belum diperlukan selama warning sudah mewakili kondisi yang tidak dapat dipastikan.

## Konsekuensi

- Kesalahan definisi yang pasti dapat menggagalkan CI sebelum deployment.
- Seluruh temuan tersedia dalam satu report.
- Dynamic validation dilaporkan secara konservatif tanpa false positive yang disengaja.
- Linter tidak menjamin branch `rules()` yang tidak dijalankan pada proses audit.
- Runtime strict mapping tetap wajib.
- Whole-application command menggunakan [discovery mechanism pada Decision 1.9](./1.9-error-definition-discovery-mechanism.md).
- Error code format dan rule ID menjadi kontrak stabil framework.
