# 1.9 Error Definition Discovery Mechanism

[← Kembali ke 1. Error Definition Framework](./README.md)

> **TL;DR:** framework menemukan enum error dan FormRequest dari autoload Composer aplikasi, bukan dari nama atau lokasi folder tertentu.

## Tujuan

Menetapkan cara menemukan seluruh target Error Definition dalam aplikasi untuk digunakan oleh linter dan generator tanpa membuat registry manual atau membebani runtime request.

## Keputusan

### Sumber Candidate

`ErrorDefinitionDiscovery` membaca bagian `autoload.psr-4` dan `autoload.classmap` dari root `composer.json` aplikasi.

PSR-4 mapping digunakan untuk membentuk candidate FQCN dari namespace prefix dan path file PHP. Classmap yang dideklarasikan aplikasi menggunakan mapping class-to-file yang telah dibuat oleh Composer.

Discovery tidak mengasumsikan folder seperti `app/Errors`, `app/Http/Requests`, atau `Modules`. Seluruh lokasi dianggap valid selama dideklarasikan pada Composer autoload aplikasi.

`autoload-dev` tidak diperiksa agar test, fixture, dan development-only class tidak masuk ke linter maupun catalog production. Dependency di dalam `vendor/` juga tidak dipindai.

### Identifikasi Target

Candidate dimuat melalui Composer autoloader dan diperiksa berdasarkan kontrak tipe:

- setiap enum yang mengimplementasikan marker interface `ErrorCode` masuk ke daftar `errorEnums`;
- setiap concrete FormRequest yang menggunakan `HasErrorDefinitions`, termasuk melalui inheritance, masuk ke daftar `formRequests`.

Discovery tidak mensyaratkan enum sudah merupakan string-backed `BackedEnum`. Target yang memakai marker interface tetapi melanggar kontrak tetap dikembalikan agar Error Definition Linter dapat melaporkan masalahnya.

Nama class seperti suffix `Error` atau `Request` tidak digunakan sebagai filter. Attribute `ErrorDefinition` juga tidak dibaca pada tahap discovery karena validasi metadata merupakan tugas Reader dan linter.

Discovery hanya mengembalikan class string. FormRequest tidak dibuat menjadi instance dan enum case tidak di-resolve.

### Hasil Discovery

Method `discover()` mengembalikan satu immutable `DiscoveryResult`:

```php
new DiscoveryResult(
    errorEnums: [
        App\Errors\UserError::class,
        Modules\SuratMasuk\Errors\SuratMasukError::class,
    ],
    formRequests: [
        App\Http\Requests\StoreUserRequest::class,
        Modules\SuratMasuk\Http\Requests\StoreSuratMasukRequest::class,
    ],
);
```

Duplicate FQCN dihapus dan setiap daftar diurutkan secara alfabetis. Urutan tidak mengikuti filesystem sehingga input yang sama selalu menghasilkan output yang deterministik.

Duplicate error code, attribute yang hilang, backing type, serta validation mapping tidak diperiksa oleh discovery. Seluruh validasi tersebut tetap menjadi tanggung jawab Error Definition Linter.

### Failure Behavior

Root `composer.json` yang tidak ditemukan, tidak valid, atau memiliki autoload path yang tidak dapat dibaca menghasilkan `DiscoveryException`.

Candidate FQCN yang tidak merujuk pada class, enum, interface, atau trait yang dapat dimuat tidak menjadi target. Interface dan trait juga diabaikan karena bukan enum error atau FormRequest. Jika Composer autoloader melempar error ketika memuat candidate, discovery membungkus penyebabnya dalam `DiscoveryException` bersama FQCN dan path terkait.

Discovery tidak mengabaikan autoload failure karena hasil yang tidak lengkap akan membuat linter dan generator terlihat berhasil meskipun sebagian definition tidak diperiksa.

### Siklus Proses

`ErrorDefinitionDiscovery` didaftarkan sebagai singleton dan menyimpan satu in-memory snapshot pada pemanggilan pertama. Pemanggilan berikutnya dalam process yang sama mengembalikan hasil yang sama tanpa mengulang filesystem scan.

Discovery hanya dijalankan oleh command, test, atau tooling yang membutuhkannya. Service Provider tidak melakukan discovery ketika aplikasi boot dan request runtime tidak bergantung pada hasil discovery.

Persistent cache atau manifest file belum digunakan. Perubahan source code akan terbaca pada process command berikutnya tanpa aturan tambahan untuk membersihkan cache lama.

### Batas Integrasi

Tidak ada config `scan_paths`, daftar enum manual, atau standalone `error-definition:discover` command. Linter dan generator menggunakan `DiscoveryResult` secara langsung.

Dependency package di dalam `vendor/` belum dapat mendaftarkan target secara otomatis. Extension point untuk package lain ditambahkan hanya ketika terdapat kebutuhan nyata; memindai seluruh vendor atau membuat provider contract sekarang tidak dipilih.

## Alasan

Composer autoload merupakan sumber resmi lokasi class PHP dalam aplikasi. Menggunakannya memungkinkan aplikasi memilih struktur minimalis, modular, atau berbasis domain tanpa mengubah framework.

Folder scanning yang di-hardcode akan bertentangan dengan Decision 1.2 dan gagal pada aplikasi yang memindahkan enum ke namespace lain. Config path manual juga mudah tertinggal ketika modul baru ditambahkan.

Marker interface dan trait menyatakan intent secara eksplisit. Naming convention seperti suffix `Error` dapat menangkap class yang bukan Error Definition dan melewatkan class dengan nama berbeda.

Scope dibatasi pada aplikasi utama agar discovery tidak memuat ribuan class dependency dan tidak mengambil Error Definition yang tidak dimiliki aplikasi. `autoload-dev` dikeluarkan karena fixture serta fake enum tidak boleh memengaruhi duplicate detection atau catalog production.

Class string dipilih agar discovery tidak menjalankan constructor FormRequest atau membaca metadata. Consumer menentukan kapan target dibuat, di-resolve, atau divalidasi.

Sorting dan deduplication membuat output stabil untuk CI dan generator. Filesystem order dapat berbeda antar-operating system dan menghasilkan diff catalog yang tidak relevan.

Fail-fast pada autoload error mencegah partial result digunakan sebagai catalog resmi. Mengabaikan class yang gagal dimuat akan menyembunyikan configuration error.

In-memory snapshot cukup karena discovery hanya digunakan selama command process. Persistent manifest membutuhkan aturan pembersihan cache dan file hasil generate baru tanpa bukti bahwa scan Composer terlalu lambat.

Standalone discovery command tidak diperlukan karena hasil class string bukan output yang digunakan pengguna akhir. Linter dan generator merupakan consumer yang memiliki output sendiri.

## Konsekuensi

- Struktur folder aplikasi tetap bebas selama mengikuti Composer autoload.
- Linter dan generator menggunakan sumber target yang sama.
- Test fixture dan dependency vendor tidak masuk hasil discovery.
- Enum yang menggunakan marker tetapi belum valid tetap dapat dilaporkan linter.
- Autoload error menggagalkan command sebelum partial result digunakan.
- Scan hanya terjadi sekali pada setiap command process.
- Perubahan `composer.json` tetap membutuhkan `composer dump-autoload` agar Composer autoloader mengenali mapping baru.
- Dukungan target dari dependency package ditunda sampai kebutuhan integrasinya jelas.
