# 1.2 Struktur Enum Error per Modul

## Tujuan

Error code dikelompokkan berdasarkan modul atau fitur yang memiliki aturan tersebut.

Prinsip utama:

> Satu enum mewakili satu modul, fitur, atau proses bisnis yang cukup jelas.

Hindari membuat satu enum global seperti:

```php
enum ApplicationError: string
{
    // seluruh error aplikasi
}
```

Karena akan cepat membesar dan sulit dipelihara.

## Contoh

```php
enum SuratMasukError: string
{
    #[ErrorDefinition(
        message: 'Nomor surat wajib diisi.',
        category: ErrorCategory::VALIDATION,
        httpStatus: 422,
        severity: ErrorSeverity::LOW,
    )]
    case NOMOR_SURAT_REQUIRED = 'SM-VAL-001';

    #[ErrorDefinition(
        message: 'Surat sudah disetujui dan tidak dapat diubah.',
        category: ErrorCategory::WORKFLOW,
        httpStatus: 409,
    )]
    case DRAFT_LOCKED = 'SM-WF-001';
}
```

## Format Error Code

```text
SM-VAL-001
│   │   └── nomor urut
│   └────── kategori error
└────────── prefix modul atau fitur
```

Contoh prefix:

```text
SM   = Surat Masuk
USR  = User
ROL  = Role
URA  = User Role Assignment
```

## Struktur Folder Minimalis

Cocok untuk aplikasi sederhana atau tim yang belum menggunakan struktur modular.

```text
app/
└── Errors/
    ├── SuratMasukError.php
    ├── UserError.php
    ├── RoleError.php
    └── UserRoleAssignmentError.php
```

Struktur ini mudah dipahami dan tidak memaksa programmer mengikuti arsitektur tertentu.

## Struktur Modular

Cocok untuk aplikasi yang sudah dikelompokkan berdasarkan modul.

```text
app/
└── Modules/
    ├── SuratMasuk/
    │   └── Errors/
    │       └── SuratMasukError.php
    │
    ├── User/
    │   └── Errors/
    │       ├── UserError.php
    │       └── UserRoleAssignmentError.php
    │
    └── Role/
        └── Errors/
            └── RoleError.php
```

## Struktur Berbasis Domain

Bisa digunakan jika aplikasi memiliki aturan bisnis yang kompleks.

```text
app/
└── Domains/
    └── IdentityAccess/
        ├── Users/
        │   └── Errors/
        │       └── UserError.php
        ├── Roles/
        │   └── Errors/
        │       └── RoleError.php
        └── Assignments/
            └── Errors/
                └── UserRoleAssignmentError.php
```

Struktur domain tidak berarti aplikasi harus menggunakan microservices. Struktur ini tetap dapat digunakan dalam Laravel monolith.

## Menentukan Pemilik Error

Patokannya bukan halaman tempat error muncul, tetapi aturan atau proses yang gagal.

Contoh pada modul pengaturan user:

```text
Email user sudah digunakan
→ UserError

User tidak ditemukan
→ UserError

Role tidak ditemukan
→ RoleError

Role sudah dimiliki user
→ UserRoleAssignmentError

Role tidak boleh diberikan kepada user tertentu
→ UserRoleAssignmentError
```

Contoh:

```php
enum UserRoleAssignmentError: string
{
    #[ErrorDefinition(
        message: 'Role tersebut sudah dimiliki oleh pengguna.',
        category: ErrorCategory::BUSINESS_RULE,
        httpStatus: 409,
    )]
    case ROLE_ALREADY_ASSIGNED = 'URA-BR-001';

    #[ErrorDefinition(
        message: 'Role tidak dapat diberikan kepada pengguna ini.',
        category: ErrorCategory::BUSINESS_RULE,
        httpStatus: 422,
    )]
    case ROLE_NOT_ASSIGNABLE = 'URA-BR-002';
}
```

## Aturan yang Disepakati

* Satu enum untuk satu modul, fitur, atau proses bisnis.
* Satu prefix tetap untuk setiap enum atau konteks error.
* Nama enum case menjelaskan makna error, bukan isi pesannya.
* Nomor error yang sudah digunakan tidak boleh digunakan ulang.
* Error dari relasi atau proses lintas modul dapat memiliki enum sendiri.
* Lokasi folder tidak ditentukan oleh framework.
* Framework tidak boleh bergantung pada struktur `Modules`, `Domains`, atau struktur folder tertentu.
* Struktur folder minimalis maupun modular sama-sama didukung.

## Keputusan

Untuk boilerplate, struktur minimalis menjadi pilihan awal yang paling mudah diikuti:

```text
app/Errors/
```

Struktur modular atau domain dapat digunakan oleh aplikasi yang memang membutuhkannya tanpa mengubah konsep Error Definition Framework.
